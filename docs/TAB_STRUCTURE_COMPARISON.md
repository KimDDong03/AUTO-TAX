# AUTO-TAX 탭 구조안 비교

> Status: decision record.
> This records the current IA decision that onboarding is not a persistent active-workspace top-level tab, while issuance and certificates stay independent top-level areas.

> 이 문서는 와이어프레임 전 단계에서 `상위 탭을 어떻게 나눌지` 비교하기 위한 문서다.  
> 비교를 거쳐 현재 기준안은 `B. 도입 준비 흡수형`으로 정리한다.

## 1. 비교 목적

- 어떤 기능을 1차 탭으로 둘지 정리한다.
- `도입 준비`, `세금계산서 발행`, `인증서`를 독립 탭으로 둘지 판단 재료를 만든다.
- 일반 사용자 앱과 플랫폼 관리자 앱의 거리감을 결정하기 위한 기준을 정리한다.

## 2. 공통 비교 대상

아래 기능들의 위치가 구조 차이를 만든다.

- 홈
- 도입 준비
- 세금계산서 발행
- 고객
- 인증서
- 설정
- 관리자

## 3. 비교 기준

- 처음 들어온 사용자가 구조를 이해하기 쉬운가
- 일상 운영자가 자주 쓰는 기능이 바로 보이는가
- 준비 단계와 운영 단계를 구분하기 쉬운가
- 인증서 도메인이 묻히지 않는가
- 탭 수가 과도하게 많아지지 않는가
- 관리자 영역이 일반 앱과 섞이지 않는가

## 4. 구조안 A: 현재 구현에 가까운 분리형

### 상위 탭

```text
홈 / 세금계산서 발행 / 고객 / 인증서 / 설정 / 도입 준비 / 관리자(조건부)
```

### 특징

- 운영 도메인을 그대로 1차 탭으로 나눈다.
- `세금계산서 발행`, `인증서`, `도입 준비`가 모두 독립 탭이다.
- 현재 구현과 가장 가깝다.

### 장점

- 자주 쓰는 운영 기능이 1차 메뉴에 직접 드러난다.
- 발행과 인증서가 다른 화면에 흡수되지 않는다.
- 도입 준비를 다시 열어보기 쉽다.
- 현재 코드 구조와 괴리가 적다.

### 리스크

- 상위 탭 수가 많다.
- 신규 사용자는 `홈`, `도입 준비`, `설정`의 차이를 바로 이해하지 못할 수 있다.
- 일상 운영 사용자는 도입 준비를 자주 쓰지 않더라도 계속 보게 된다.

### 이 안이 유리한 상황

- 운영 사용자가 기능을 직접 찾는 빈도가 높을 때
- 인증서가 별도 핵심 도메인으로 보이길 원할 때
- 현재 구조와의 연속성이 중요할 때

### 확인 질문

- `도입 준비`를 상시 1차 탭으로 노출할 필요가 있는가
- 탭 수가 많아 보여도 사용성이 유지되는가

## 5. 구조안 B: 도입 준비 흡수형

### 상위 탭

```text
홈 / 세금계산서 발행 / 고객 / 인증서 / 설정 / 관리자(조건부)
```

### 도입 준비 위치

- 홈 안 준비 허브
- 또는 설정/홈에서 다시 열 수 있는 보조 entry

### 특징

- 1차 메뉴에서 `도입 준비`를 빼고 홈 또는 별도 진입점 안에 흡수한다.
- 운영 단계 탭은 유지한다.

### 장점

- 상위 구조가 간결해진다.
- 홈이 `현재 상태 + 다음 행동` 허브로 더 강해진다.
- 신규와 기존 사용자의 진입점을 홈 중심으로 통일하기 쉽다.

### 리스크

- 도입 준비가 약해 보일 수 있다.
- 초기 세팅 사용자가 준비 흐름을 다시 찾기 어려울 수 있다.
- 홈의 역할이 커지면서 정보 밀도가 과해질 수 있다.

### 이 안이 유리한 상황

- 홈을 강한 운영 허브로 설계하고 싶을 때
- 신규보다 반복 운영 사용자의 빈도가 더 중요할 때
- 상위 탭 수를 줄이고 싶을 때

### 확인 질문

- 도입 준비를 홈 안에 넣으면 존재감이 충분한가
- 준비 흐름 재진입이 어려워지지 않는가

### 현재 기준안에서의 채택 방식

- 이 문서의 최종 기준안은 `B`다.
- 다만 `인증서`와 `세금계산서 발행`은 독립 1차 탭으로 유지한다.
- `도입 준비`는 상시 1차 탭이 아니라 `홈 안 준비 허브 + 설정에서 다시 열기` 구조로 둔다.
- 완료 후에는 기본적으로 숨기고, 필요하면 사용자가 다시 열 수 있게 한다.

## 6. 구조안 C: 인증서 통합형

### 상위 탭

```text
홈 / 세금계산서 발행 / 고객 / 설정 / 도입 준비 / 관리자(조건부)
```

### 인증서 위치

- 설정 하위 섹션
- 또는 고객 하위 컨텍스트

### 특징

- `인증서`를 독립 1차 탭에서 빼고 다른 도메인 아래 둔다.
- 상위 탭 수를 줄이려는 방향이다.

### 장점

- 상위 메뉴 수를 줄일 수 있다.
- 설정/고객과 연관된 보조 기능처럼 보이게 만들 수 있다.

### 리스크

- 인증서가 실제보다 보조 기능처럼 읽힐 수 있다.
- 고객 운영과 발행 준비에서 차지하는 중요도가 흐려질 수 있다.
- helper 상태, 로컬 인증서, 연결 상태, 준비/결제 상태를 한 하위 섹션에 담아야 해서 밀도가 높아질 수 있다.

### 이 안이 유리한 상황

- 상위 메뉴 수를 강하게 줄여야 할 때
- 인증서를 독립 도메인보다 보조 관리 기능으로 보려 할 때

### 확인 질문

- 인증서가 지금 제품에서 보조 기능인가, 핵심 도메인인가
- 설정 또는 고객 하위로 들어갔을 때 탐색성이 유지되는가

## 7. 구조안 D: 운영 집중 축소형

### 상위 탭

```text
홈 / 작업 / 고객 / 설정 / 관리자(조건부)
```

### 내부 재배치 예시

- `작업`
  - 세금계산서 발행
  - 미매칭 메일
  - 첫 메일 동기화 결과
- `고객`
  - 고객 관리
  - 인증서 연결/상태 일부 포함
- `설정`
  - 메일 연결
  - 발행 설정
  - 헬퍼 상태
  - 도입 준비 entry

### 특징

- 상위 탭 수를 크게 줄이고 운영 관점으로 다시 묶는다.
- 도메인 분리보다 사용 빈도와 작업 맥락 중심으로 구조를 바꾼다.

### 장점

- 상위 구조는 가장 단순해진다.
- 운영자가 “지금 해야 할 일” 기준으로 움직이기 쉽다.

### 리스크

- 현재 구현과 차이가 크다.
- 기능 경계가 흐려질 수 있다.
- 고객/인증서/예외 메일/발행이 어디까지 한 탭인지 합의가 더 많이 필요하다.

### 이 안이 유리한 상황

- 제품을 `업무 도구`로 더 강하게 재해석하려 할 때
- 상위 탭 단순화가 최우선일 때

### 확인 질문

- 기능 경계보다 작업 흐름을 더 우선할 것인가
- 구조 단순화가 정보 과밀을 만들지 않는가

## 9. 최종 결정

### 결정 1. 도입 준비는 상시 1차 탭으로 두지 않는다

- `도입 준비`는 제품의 핵심 운영 탭이라기보다 초기 진입과 재설정 지원용 허브로 본다.
- 상위 1차 탭에서는 제외하고 `홈`에서 현재 진행 상태와 다음 행동을 강하게 보여준다.
- 재진입 경로는 `홈`과 `설정`에 둔다.
- 완료 후에는 자동으로 숨기고, 필요하면 사용자가 다시 열 수 있게 한다.

### 결정 2. 인증서는 독립 도메인으로 유지한다

- `인증서`는 단순 보조 설정이 아니라 고객 운영, 발행 준비, helper 상태와 직접 연결되는 핵심 도메인이다.
- 따라서 `설정` 하위 보조 섹션으로 흡수하지 않고 독립 1차 탭으로 유지한다.

### 결정 3. 세금계산서 발행은 독립 탭으로 유지한다

- `세금계산서 발행`은 홈의 요약 카드가 아니라 실제 작업과 처리 흐름이 집중되는 화면이다.
- 따라서 `작업` 같은 통합 탭으로 묶기보다 독립 1차 탭으로 남기는 편이 탐색성과 작업 속도 모두에 유리하다.

### 결정 4. 관리자 영역은 일반 사용자 앱과 분리한다

- `관리자`는 일반 사용자 업무 흐름과 목적이 다르므로 같은 레벨에서 섞어 보이지 않게 한다.
- 권한이 있는 경우에만 조건부로 노출하는 별도 영역으로 유지한다.

## 10. 최종 기준안 요약

### 상위 탭

```text
홈 / 세금계산서 발행 / 고객 / 인증서 / 설정 / 관리자(조건부)
```

### 도입 준비 노출 원칙

- 기본 위치는 `홈 안 준비 허브`다.
- `설정`에서 `도입 준비 다시 열기`를 제공한다.
- 완료 후에는 기본 숨김으로 본다.
- 필요 시 사용자가 수동으로 다시 노출할 수 있게 한다.

### 이 기준안을 고른 이유

- 신규 사용자는 `홈` 중심 진입이 가장 이해하기 쉽다.
- 반복 운영 사용자는 `발행 / 고객 / 인증서`를 더 자주 오간다.
- `도입 준비`는 중요하지만 상시 운영 탭이라기보다 초기 진입용 보조 허브에 가깝다.
- `인증서`는 제품에서 독립 도메인으로 유지해야 한다.
- 상위 탭 수는 줄이되, 핵심 도메인까지 뭉개지는 수준의 단순화는 피한다.

## 11. 와이어프레임 진행 기준

- `홈`은 상태 요약과 다음 행동, 그리고 도입 준비 재진입 허브를 맡는다.
- `세금계산서 발행`, `고객`, `인증서`, `설정`은 각각 독립 1차 화면으로 와이어프레임을 진행한다.
- `도입 준비`는 별도 상시 탭이 아니라 홈/설정에서 진입하는 흐름으로 설계한다.
- `관리자`는 일반 사용자 앱과 분리된 조건부 영역으로만 다룬다.
