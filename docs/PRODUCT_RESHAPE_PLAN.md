# AUTO-TAX Product Reshape Plan

이 문서는 현재 제품을 `기능 묶음 중심 운영 시스템`에서 `사용자 목표 중심 제품`으로 재정렬하기 위한 실행 기준이다.

현재 반영 상태:

- `도입 준비` 탭 신설
- `초기 등록` 단독 탭 제거 및 `도입 준비` 편입 시작
- `오늘 작업` 상단 `긴급 예외` 카드 구조 도입
- 고객 상세 상단 `발행 사유 + 해결 액션` 체크리스트 도입
- `인증서 관리` 기본 진입을 `조치 필요` 필터 중심으로 전환

## 1. 문제 정의

현재 AUTO-TAX는 기능 자체는 충분하지만, 화면 구조가 사용자 목표보다 내부 시스템 구성을 더 많이 닮아 있다.

대표 증상:

- 사용자는 첫 성공까지의 순서를 제품이 안내해주기보다 직접 해석해야 한다.
- `초기 등록`, `시스템설정`, `공동인증서`가 하나의 도입 흐름으로 보이지 않는다.
- `오늘 작업`은 상태는 잘 보이지만 긴급도 순으로 읽히지 않는다.
- 고객 화면은 `발행 가능/불가` 상태는 보여주지만, 막힌 이유와 바로 해결 액션이 약하다.

핵심 판단:

- 지금 필요한 것은 새 기능 추가보다 정보 구조 재편이다.
- 시각 언어는 유지하고, 사용자 흐름과 우선순위를 다시 세운다.

## 2. 목표

이번 리쉐이프의 목표는 아래 세 가지다.

1. 첫 사용자가 `첫 발행`까지 가는 경로를 제품이 직접 안내한다.
2. 일상 운영자는 `지금 무엇을 처리해야 하는지`를 즉시 파악한다.
3. 고객별 막힘 원인을 화면이 먼저 설명하고, 해결 액션까지 붙인다.

## 3. 확정한 개선 항목

### A. IA 재구성

탭/메뉴는 기능명보다 사용자 목표 기준으로 재정렬한다.

기존:

- 오늘 작업
- 고객관리
- 공동인증서
- 초기 등록
- 시스템설정

목표:

- 오늘 작업
- 도입 준비
- 고객 운영
- 인증서 관리
- 작업공간 설정
- 플랫폼 관리자

원칙:

- `도입 준비`는 초반 셋업과 초기 등록을 묶는 가이드형 진입점이다.
- `작업공간 설정`은 상세 설정을 다루는 별도 화면으로 유지한다.
- `플랫폼 관리자`는 현행 분리 구조를 유지한다.

### B. 오늘 작업 재정렬

현재 `오늘 작업`은 상태 요약은 좋지만, 긴급도와 행동 우선순위가 분산되어 있다.

개선 방향:

- 최상단에 `긴급 예외 바`를 둔다.
- `긴급 예외 > 지금 처리할 건 > 최근 처리` 순으로 읽히게 만든다.
- 중복 경고는 hero, panel, list에 흩어놓지 않고 상단에서 먼저 집계한다.

### C. 도입 준비 흐름 신설

초기 도입 과정은 다음 8단계로 정리한다.

1. 메일 연결
2. 발행 기본값 입력
3. 로컬 헬퍼 준비
4. 고객 초기 등록
5. 인증서 연결 마무리
6. 첫 메일 동기화
7. 미매칭 메일 예외 처리
8. 첫 발행 확인

원칙:

- 각 단계는 `상태`, `왜 필요한지`, `바로 실행 액션`, `성공 조건`을 함께 보여준다.
- 기술 필드는 그대로 두더라도, 첫 노출은 업무 언어로 번역한다.
- `메일 연결`과 `메일 동기화`는 분리한다. 먼저 연결 테스트만 하고, 실제 동기화는 고객/인증서 준비 후 실행한다.
- 엑셀 양식은 현재 PC의 공동인증서 목록 기준이므로 `로컬 헬퍼 준비`를 `고객 초기 등록`보다 앞에 둔다.
- `미매칭 메일`은 메인 onboarding 단계가 아니라 첫 동기화 뒤에 처리하는 예외 단계로 둔다.

### D. 고객 운영 개선

고객 화면은 고객별 상태 설명을 더 직접적으로 바꾼다.

개선 방향:

- 고객 상세 상단에 `발행 가능 / 발행 불가`를 명확히 표시한다.
- 불가일 경우 사유를 체크리스트로 나열한다.
- 각 사유 옆에 해결 버튼을 붙인다.

예시:

- 팝빌 가입 필요
- 전자세금용 인증서 미등록
- 인증서 만료 예정
- 주소 매칭 확인 필요

### E. 인증서 관리 재정렬

인증서 화면은 인증서 자체 목록보다 `조치가 필요한 고객/인증서` 기준으로 정렬한다.

개선 방향:

- 기본 진입 필터를 `조치 필요`로 둔다.
- `갱신 필요`, `결제 가능`, `미연결`, `정상 연결 완료` 기준으로 나눈다.

## 4. 비목표

이번 리쉐이프에서 하지 않는 것:

- 새로운 발행 기능 추가
- 역할 체계 재설계
- 시각 언어 전면 교체
- 플랫폼 관리자 화면 대개편

## 5. 1차 개발 범위

이번 작업에서 바로 시작할 범위는 아래로 제한한다.

1. 메뉴/탭 문구를 사용자 목표 중심으로 정리한다.
2. `도입 준비` 탭을 신설한다.
3. `초기 등록` 기능을 `도입 준비` 안으로 편입한다.
4. `작업공간 설정` 탭은 상세 설정 화면으로 유지한다.

이 단계는 전체 리쉐이프의 기반 공사다.
`오늘 작업`, `고객 운영`, `인증서 관리`의 상세 개편은 이후 단계에서 이어간다.

## 6. 구현 순서

### Phase 1

- `App.tsx` 탭 구조 개편
- `도입 준비` 탭 추가
- `초기 등록` 단독 탭 제거
- 기존 `SettingsTab`, `InitialRegistrationTab`를 새 흐름에서 재배치

### Phase 2

- `오늘 작업` 상단 긴급 예외 바 도입
- 중복 상태 신호 정리

### Phase 3

- 고객 상세 상단에 막힘 사유 + 해결 액션 추가

### Phase 4

- 인증서 화면 기본 필터와 정렬 기준 재구성

## 7. 주요 파일

- `web/src/App.tsx`
- `web/src/styles.css`
- `web/src/features/settings/SettingsTab.tsx`
- `web/src/features/initial-registration/InitialRegistrationTab.tsx`
- `web/src/features/customers/CustomersTab.tsx`
- `web/src/features/certificates/CertificatesTab.tsx`

## 8. 실행 메모

- 문서상의 방향과 실제 구현이 충돌하면, 우선 `도입 준비` 흐름을 기준으로 판단한다.
- 기존 디자인 시스템의 토큰과 패널 구조는 유지한다.
- 1차 구현은 `새 흐름 진입점 만들기`가 목적이지, 모든 세부 화면을 완성하는 것이 목적이 아니다.
