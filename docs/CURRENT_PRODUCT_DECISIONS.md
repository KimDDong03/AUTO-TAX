# AUTO-TAX 현재 제품 결정

이 문서는 2026-03-26 기준으로 개발 진행 전에 합의한 제품 방향을 짧게 고정해두는 용도다.

## 1. 서비스 정의

- AUTO-TAX의 직접 고객은 `태양광 회사`다.
- 태양광 회사가 AUTO-TAX 안에서 자기 `관리 고객`을 관리한다.
- `관리 고객`은 실제 세금계산서 발행 대상이지만, AUTO-TAX 로그인 사용자는 아니다.
- 따라서 AUTO-TAX는 고객 포털이 아니라 `태양광 회사 내부 운영 시스템`이다.

## 2. 로그인 대상과 권한

### 2-1. 로그인 가능한 주체

- `운영사 슈퍼관리자`
- `고객사 owner`
- `고객사 member`

`관리 고객`은 계정을 만들지 않고 로그인하지 않는다.

### 2-2. 로그인 방식

- `운영사 슈퍼관리자`만 이메일 로그인
- `고객사 owner/member`는 `로그인 아이디 + 비밀번호`
- 고객사 사용자는 이메일 계정을 로그인 기준으로 쓰지 않는다.

### 2-3. 권한 원칙

- `운영사 슈퍼관리자`
  - 고객사 상태 확인
  - 장애 대응
  - 설정 보조
  - 고객사 화면과 분리된 `운영자 전용 페이지` 사용
  - 내부 운영용 최고 권한
- `고객사 owner`
  - 구독/결제
  - owner 변경
  - 작업공간 비활성화/삭제
  - 고객사 내부 사용자 관리
  - member 임시 비밀번호 재설정
- `고객사 member`
  - 작업공간 설정
  - 관리 고객 등록/수정
  - 초안 검수와 발행
  - 인증서 상태 확인 및 발행 준비 보조 실행

owner 계정 임시 비밀번호 재설정은 운영사 슈퍼관리자가 담당한다.

1차 버전에서는 `admin`, `operator`, `viewer`를 두지 않는다.

## 3. 가입과 온보딩

- 공개 회원가입은 열지 않는다.
- 운영사가 고객사 작업공간과 첫 `owner` 계정을 개통한다.
- 초기 온보딩은 운영사가 잡고, 이후 일상 운영은 고객사가 맡는다.
- 초기 `관리 고객` 데이터는 고객사가 화면에서 수작업으로 입력하는 것을 기준으로 본다.

## 4. 설정과 비밀값 경계

### 3-1. 서버 env에만 두는 값

- Supabase 서버 키
- 팝빌 `LinkID`
- 팝빌 `SecretKey`
- 팝빌 파트너 결제용 사업자번호

이 값들은 브라우저로 내려가면 안 된다.

### 3-2. 고객사 작업공간 설정에 두는 값

- 메일 계정과 앱 비밀번호
- 메일 동기화/발행 스케줄
- 팝빌 사용자 ID 접두어
- 신규 고객 기본 비밀번호
- 운영 담당자 이름/이메일/연락처

즉 고객사마다 달라지는 운영값은 env가 아니라 작업공간 데이터로 관리한다.

## 5. 운영 방식

- 구독/결제는 1차 버전에서 제품 안 UI로 만들지 않고, 운영 절차로 처리한다.
- 기본 자동 흐름은 `월 자동 실행일 자동 메일 동기화 -> 고객 매칭 -> 자동발행 고객 즉시 발행`으로 본다.
- `검수 후 발행` 고객은 초안만 만들고 작업공간에서 수동 발행한다.
- 사용자는 언제든 `메일 동기화`, `개별 발행`, `전체 발행`을 수동 실행할 수 있다.
- 매칭 실패, 파싱 실패, 자동 발행 실패 같은 예외는 운영 메일 알림 대상으로 본다.

## 6. 배치 구조

- 메일 수집, 자동발행, 인증서 점검은 Vercel 요청 처리와 분리된 `별도 Node 워커` 구조를 기준으로 본다.
- 작업 상태와 이력은 `Supabase 영속 큐`에 저장한다.
- 초기 무료 운영 단계에서는 `Supabase pg_cron + Edge Functions`가 정기 트리거를 맡는다.
- 이때 크론은 무거운 작업을 직접 처리하지 않고 `job_queue`에 due 작업을 넣는 `dispatch`까지만 맡는다.
- 월 정기 자동 처리는 `고객사 작업공간에서 정한 월 실행일/시각`에 `mail-sync` 작업을 생성하는 방식으로 간다.
- `mail-sync` 작업이 자동발행 고객의 초안을 만들면 같은 큐에서 `auto-issue` 작업까지 이어서 처리한다.
- 큐 작업은 타입별 재시도 횟수 안에서 자동 재시도하고, 끝까지 실패한 건만 최종 실패와 알림 대상으로 남긴다.
- 실제 긴 처리나 재시도 제어가 필요한 작업은 이후 `별도 Node 워커`로 분리할 수 있게 설계한다.
- 메일 수집은 `자동 + 수동` 둘 다 지원한다.
- 메일/자동발행/인증서 작업은 같은 `job_queue`에서 job type으로 구분한다.

## 7. 인증서 범위

- 1차 버전은 `공동인증서 갱신 보조`까지 포함하지 않는다.
- 대신 `고객사 전용 1대 PC`의 로컬 공동인증서를 이용한 발행 준비는 포함한다.
- 전용 PC는 초기 발행 준비와 예외 대응에 사용한다.
- 월별 정기 전자세금계산서 발행은 준비 완료 후 `팝빌 API`로 서버에서 처리한다.

## 8. 용어

- `고객사`: AUTO-TAX를 구독하는 태양광 회사
- `사용자`: 고객사 내부 사용자
- `관리 고객`: 고객사가 관리하는 실제 발행 대상 고객
- `작업공간`: 고객사별 독립 데이터 공간
