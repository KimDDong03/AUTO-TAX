# AUTO-TAX 와이어프레임 전 단계 기능명세

## 문서 목적

- 이 문서는 `2026-04-23` 현재 AUTO-TAX 구현을 기준으로, 와이어프레임 설계 전에 필요한 기능 판단 재료를 정리한 문서다.
- 목적은 화면 구조 결론을 내리는 것이 아니라, 구조를 결정할 때 반드시 고려해야 할 사용자 목적, 정보, 액션, 상태 분기를 드러내는 데 있다.
- 이 문서는 `리스트가 맞다`, `드로어가 맞다`, `모달이 맞다` 같은 구조 결론을 의도적으로 내리지 않는다.
- 기준 소스는 현재 체크아웃의 `web/src/App.tsx`, `web/src/features/*`, `server/src/routes/*`, `docs/IMPLEMENTATION.md`, `docs/IMPLEMENTATION_STATUS.md`다.

## 사용 방법

- 각 기능 섹션은 먼저 `이 기능이 왜 존재하는지`를 확인한다.
- 그 다음 `사용자가 무엇을 끝내려는지`, `무엇을 반드시 봐야 하는지`, `무엇을 할 수 있어야 하는지`를 확인한다.
- 마지막으로 `구조에 영향을 주는 사실`과 `열어둘 쟁점`을 보고 화면 분리, 통합, 단계화 여부를 고민한다.

## 전역 범위

- 현재 사용자 앱의 주요 화면 단위는 `도입 준비`, `홈`, `발행`, `고객`, `인증서`, `설정`이다.
- `관리자(ops)`는 일반 사용자 앱과 목적이 다른 플랫폼 운영 화면이다.
- 공개 루트 `/`는 마케팅 랜딩이 아니라 고객사 전용 로그인 포털이다.

## 전역 상태와 용어

### 초안 상태

- `review`: 검수 대기
- `scheduled`: 발행 예약 대기
- `issuing`: 발행 중
- `issued`: 발행 완료
- `failed`: 발행 실패

### 메일 상태

- `pending`: 수집 후 처리 대기
- `parsed`: 파싱 완료
- `failed`: 파싱 실패
- `unmatched`: 고객 미매칭
- `duplicate`: 중복 의심
- `ignored`: 운영상 제외

### 고객 상태

- `popbillState`: `pending`, `joined`, `failed`
- `issueMode`: `review`, `auto`
- 고객은 발행 준비 상태, 팝빌 연결 상태, 인증서 상태를 함께 가진다.

### 권한 상태

- DB 역할은 `owner`, `admin`, `operator`, `viewer`가 존재한다.
- 현재 UX는 대부분 `owner`와 `non-owner member` 중심으로 동작한다.
- `ops`는 `platform admin` 전용이다.

### 로컬 헬퍼/자동화 상태

- 로컬 공동인증서 관련 흐름은 Windows helper와 renewal agent 상태에 영향을 받는다.
- helper/agent 연결 여부, 읽은 인증서 수, 업그레이드 필요 여부, 진단 작업 상태가 화면 상태에 영향을 준다.

## 1. 공개 로그인

### 기능 목적

- 계약이 완료된 고객사 사용자가 서비스에 진입하고 작업공간 세션을 시작한다.

### 주요 사용자

- 고객사 owner
- 고객사 내부 운영자

### 진입 맥락

- 사용자는 외부에서 직접 서비스 URL로 진입한다.
- 아직 작업공간 앱 내부 상태를 보지 못한 상태에서 처음 접한다.

### 사용자가 완료하려는 일

- 자신의 계정으로 로그인한다.
- 접근 가능한 작업공간으로 진입한다.

### 화면에 반드시 있어야 하는 정보

- 로그인 계정 입력 위치
- 비밀번호 입력 위치
- 로그인 실패 시 이해 가능한 오류 메시지
- 이 화면이 누구를 위한 화면인지 설명하는 최소한의 안내

### 반드시 가능한 액션

- 로그인
- 오류 확인 후 재입력

### 상태/예외

- 기본 입력 상태
- 로그인 진행 중 상태
- 인증 실패 상태
- 성공 후 이동 상태

### 다른 기능과의 관계

- 성공 시 작업공간 bootstrap으로 이어진다.
- 실패 시 아직 다른 기능으로 진입하지 못한다.

### 구조에 영향을 주는 사실

- 사용자는 이 화면에서 한 가지 일만 끝내면 된다.
- 진입 전 사용자는 서비스 내부 구조를 모른다.
- 오류 메시지의 명확성이 재시도 성공률에 직접 영향을 준다.

### 열어둘 쟁점

- 로그인 외 보조 정보의 밀도
- 안내 텍스트의 범위
- 지원/문의 정보 노출 수준

## 2. 작업공간 초기 진입과 전역 셸

### 기능 목적

- 로그인 후 사용자가 현재 작업공간의 상태를 불러오고, 권한에 맞는 업무 화면으로 진입할 수 있게 한다.

### 주요 사용자

- 작업공간 owner
- 작업공간 member
- platform admin

### 진입 맥락

- 공개 로그인 성공 직후 진입한다.
- 사용자는 당장 어떤 일을 먼저 해야 하는지 모를 수 있다.

### 사용자가 완료하려는 일

- 현재 작업공간이 어떤 상태인지 파악한다.
- 지금 해야 할 업무 화면으로 이동한다.

### 화면에 반드시 있어야 하는 정보

- 현재 작업공간 정보
- 현재 사용자 정보
- 접근 가능한 주요 화면 목록
- 작업공간이 테스트/운영 중 어느 모드인지 알 수 있는 정보

### 반드시 가능한 액션

- 주요 화면 간 이동
- 작업공간 맥락을 유지한 채 데이터 조회

### 상태/예외

- 정상 bootstrap 완료
- bootstrap 로딩 중
- 활성 작업공간 없음
- platform admin이지만 일반 작업공간가 아직 선택되지 않은 상태

### 다른 기능과의 관계

- 홈, 도입 준비, 고객, 발행, 인증서, 설정, 관리자 화면의 공통 진입점이다.

### 구조에 영향을 주는 사실

- 이 영역은 독립 업무를 수행하는 곳이라기보다 다른 업무 화면으로 이동하기 위한 전역 문맥 역할이 크다.
- 권한에 따라 노출되는 메뉴가 달라진다.
- 일반 사용자 앱과 관리자 앱의 목적이 다르다.

### 열어둘 쟁점

- 전역 네비게이션의 정보량
- 사용자 앱과 관리자 앱의 분리 수준
- 작업공간 상태 표시 밀도

## 3. 도입 준비 허브

### 기능 목적

- 신규 작업공간 사용자가 서비스 사용 준비를 순서대로 마칠 수 있게 한다.

### 주요 사용자

- 신규 owner
- 초기 세팅을 맡은 운영자

### 진입 맥락

- 로그인 직후 아직 메일, 고객, 인증서, 발행 흐름이 준비되지 않은 상태에서 진입한다.
- 홈이나 설정에서 다시 열어볼 수도 있다.

### 사용자가 완료하려는 일

- 서비스 운영을 시작하기 전 필요한 선행 작업을 빠짐없이 끝낸다.

### 화면에 반드시 있어야 하는 정보

- 현재 단계 목록
- 각 단계의 완료 여부
- 현재 막힌 단계와 막힌 이유
- 다음에 해야 할 일

### 반드시 가능한 액션

- 특정 단계 열기
- 현재 단계에서 필요한 설정/업로드/동기화 실행
- 완료 후 다음 단계로 이동

### 상태/예외

- 아직 아무 것도 시작하지 않은 상태
- 일부 단계만 완료된 상태
- 특정 단계가 선행 조건 부족으로 막힌 상태
- 모든 핵심 단계가 완료된 상태

### 다른 기능과의 관계

- 설정, 초기 등록, 인증서, 메일 동기화, 발행 확인으로 이어진다.

### 구조에 영향을 주는 사실

- 단계 간 선후관계가 강하다.
- 사용자는 한 번에 모든 정보를 보기보다 현재 단계와 다음 단계를 우선적으로 본다.
- 동일한 도입 흐름이 홈, 설정, 초기 등록과 연결된다.

### 열어둘 쟁점

- 전체 단계 노출 수준
- 완료된 단계의 재진입 방식
- 단계별 상세를 같은 화면에서 풀지, 별도 진입으로 나눌지

## 4. 고객 초기 등록

### 기능 목적

- 운영 시작 전 필요한 고객 기본 데이터를 빠르게 등록하고, workbook 기반 초기 세팅을 마무리한다.

### 주요 사용자

- 신규 owner
- 초기 세팅 담당자

### 진입 맥락

- 도입 준비 중 `고객 초기 등록` 단계에서 진입한다.
- 로컬 전자세금용 인증서를 읽은 뒤 양식을 받아 진행하는 흐름이 현재 구현 기준이다.

### 사용자가 완료하려는 일

- 등록 양식을 받는다.
- 파일을 업로드해 반영 가능한 고객과 문제 행을 확인한다.
- 반영을 실행해 고객 등록을 완료한다.

### 화면에 반드시 있어야 하는 정보

- 현재 등록 단계
- 다운로드/업로드/반영 상태
- 업로드한 파일명
- 반영 가능 건수와 막힌 건수
- 막힌 사유
- 후속 인증서 등록 대기 수

### 반드시 가능한 액션

- 양식 다운로드
- 파일 업로드
- 반영 실행
- 실패 건 재확인
- 후속 인증서 등록 이어가기

### 상태/예외

- 헬퍼 미준비
- 전자세금용 인증서 없음
- 파일 미선택
- preview 준비 완료
- 반영 가능 건 존재
- blocked row 존재
- 반영 완료
- 후속 인증서 등록 대기

### 다른 기능과의 관계

- 인증서 관리와 직접 이어진다.
- 고객 데이터는 이후 고객 관리, 발행, 메일 매칭의 기반이 된다.

### 구조에 영향을 주는 사실

- 사용자는 한 번에 한 단계씩 처리하지만, 각 단계마다 확인해야 할 수치와 예외가 존재한다.
- 파일 업로드 전과 후의 정보 밀도가 크게 달라진다.
- 성공 케이스와 문제 행 처리 케이스가 한 흐름 안에 같이 존재한다.

### 열어둘 쟁점

- 단계 진행 정보와 상세 결과 정보의 상대적 비중
- blocked row 정보를 본문에 둘지 별도 확인 흐름으로 분리할지
- 후속 인증서 등록을 같은 연속 흐름으로 다룰지 분리할지

## 5. 첫 메일 동기화와 예외 메일 처리

### 기능 목적

- 첫 운영 시점에 실제 메일을 읽어 초안을 만들고, 자동 매칭에서 남은 예외를 정리한다.

### 주요 사용자

- owner
- 작업공간 운영자

### 진입 맥락

- 고객과 인증서 준비가 끝난 뒤 첫 운영 검증으로 진입한다.
- 도입 준비 안에서도 이어지고, 운영 중에는 홈/발행 흐름과 연결된다.

### 사용자가 완료하려는 일

- 메일 동기화를 실행한다.
- 초안 생성 여부를 확인한다.
- 미매칭 메일이나 실패 메일을 정리한다.

### 화면에 반드시 있어야 하는 정보

- 메일 동기화 실행 가능 여부
- 동기화 결과
- 미매칭 메일 목록
- 메일에서 읽은 주소, 발전소명, 금액, 정산월
- 어떤 이유로 자동 매칭이 안 됐는지에 대한 단서

### 반드시 가능한 액션

- 첫 메일 동기화 실행
- 개별 메일 재처리
- 예외 메일 기반 빠른 고객 등록
- 미매칭 메일 확인

### 상태/예외

- 첫 동기화 전
- 동기화 진행 중
- 초안 생성 성공
- 미매칭 메일 존재
- 파싱 실패
- 중복 메일 존재
- 더 이상 처리할 예외 없음

### 다른 기능과의 관계

- 고객 관리와 강하게 연결된다.
- 발행 콘솔로 자연스럽게 이어진다.
- 도입 준비의 마지막 검증 흐름 일부다.

### 구조에 영향을 주는 사실

- 사용자는 메일 원문보다 `무엇이 문제인지`와 `어떤 고객과 연결할지`를 먼저 판단한다.
- 예외 처리는 단일 액션보다 비교와 확인이 많은 업무다.
- 메일 정보와 고객 정보가 동시에 필요하다.

### 열어둘 쟁점

- 미매칭 메일 확인과 빠른 고객 등록의 결합 수준
- 첫 동기화 결과를 도입 준비 안에서 다룰지 독립 운영 화면으로 넘길지

## 6. 홈 대시보드

### 기능 목적

- 사용자가 지금 처리해야 할 운영 우선순위를 빠르게 파악하고 다음 액션으로 이동하게 한다.

### 주요 사용자

- owner
- 작업공간 운영자

### 진입 맥락

- 일상 운영 시작 시 가장 먼저 들어오는 화면이다.
- 도입 준비가 남아 있더라도 홈에서 현재 상태를 다시 확인한다.

### 사용자가 완료하려는 일

- 오늘 우선 처리할 항목을 파악한다.
- 발행, 예외, 고객 상태 확인 등 다음 업무로 이동한다.

### 화면에 반드시 있어야 하는 정보

- 온보딩 배너 또는 준비 상태
- 우선순위 카드
- 검수 대기 초안
- 최근 수신 메일
- 최근 발행 이력
- 현재 작업공간과 사용자 맥락

### 반드시 가능한 액션

- 우선순위 항목 열기
- 검수 초안 직접 발행
- 최근 메일 재처리
- 최근 발행 상세 보기

### 상태/예외

- 아직 도입 준비 미완료
- 검수 대기 초안 존재
- 미매칭/예외 메일 존재
- 최근 활동 없음
- 정상 운영 상태

### 다른 기능과의 관계

- 발행 콘솔, 고객 관리, 도입 준비, 설정으로 이동한다.

### 구조에 영향을 주는 사실

- 이 화면의 핵심은 업무 수행보다 업무 우선순위 판단이다.
- 서로 성격이 다른 정보가 같이 존재한다.
- 사용자는 요약을 본 뒤 다른 화면으로 자주 이동한다.

### 열어둘 쟁점

- 요약과 작업 가능 항목의 비중
- 최근 활동과 우선순위 카드의 상대적 중요도
- 도입 준비 정보의 지속 노출 수준

## 7. 발행 콘솔

### 기능 목적

- 사용자가 발행 대기 초안, 발행 실패 건, 미매칭 메일을 보고 실제 발행 처리와 예외 정리를 끝낸다.

### 주요 사용자

- owner
- 작업공간 운영자

### 진입 맥락

- 홈에서 검수 대기나 예외 메일을 보고 들어온다.
- 운영자가 오늘 발행 업무를 처리하려고 직접 진입할 수도 있다.

### 사용자가 완료하려는 일

- 검수 대기 초안을 직접 발행한다.
- 실패 건을 다시 확인한다.
- 미매칭 메일을 후속 처리한다.
- 발행 완료 건을 확인하거나 취소 후 다시 열어본다.

### 화면에 반드시 있어야 하는 정보

- 상태별 필터
- 초안 목록
- 미매칭 메일 목록
- 고객명, 품목, 정산월, 금액, 발행 상태
- 실패 사유
- 발행 상세 정보
- 팝빌 확인번호/관리번호 등 후속 확인 정보

### 반드시 가능한 액션

- 개별 직접 발행
- 검수 건 일괄 직접 발행
- 초안 보기
- 인쇄
- 팝빌 정보 확인
- 발행 취소
- 미매칭 메일 확인
- 메일 재처리

### 상태/예외

- 검수 대기 없음
- 자동 대기만 존재
- 발행 중 존재
- 발행 실패 존재
- 미매칭 메일 존재
- 선택된 항목 없음

### 다른 기능과의 관계

- 고객 관리와 연결된다.
- 홈과 상호 왕복된다.
- 팝빌 확인 흐름과 연결된다.

### 구조에 영향을 주는 사실

- 사용자는 여러 건을 비교한 뒤 하나를 선택해 상세를 확인한다.
- 상세를 본 뒤 다시 목록으로 돌아오는 일이 반복된다.
- 초안과 미매칭 메일이 같은 운영 맥락 안에서 다뤄진다.
- 요약, 목록, 상세 정보의 밀도가 모두 높다.

### 열어둘 쟁점

- 초안과 미매칭 메일을 같은 흐름으로 둘지 분리할지
- 요약 정보와 실무 처리 영역의 비중
- 일괄 처리와 개별 처리 노출 방식

## 8. 고객 관리

### 기능 목적

- 사용자가 등록된 고객의 현재 준비 상태를 파악하고 고객 정보와 연동 상태를 관리한다.

### 주요 사용자

- owner
- 작업공간 운영자

### 진입 맥락

- 고객 등록 이후 운영 과정에서 반복 진입한다.
- 홈의 막힘 고객, 인증서 상태, 발행 준비 문제와 연결되어 들어온다.

### 사용자가 완료하려는 일

- 고객을 찾는다.
- 고객을 추가하거나 수정한다.
- 고객별 팝빌/인증서/발행 상태를 확인한다.
- 필요한 후속 액션을 수행한다.

### 화면에 반드시 있어야 하는 정보

- 검색창
- 상태 필터
- 고객 요약 수치
- 고객 목록
- 고객별 팝빌 상태, 인증서 상태, 발행 방식, 다음 조치
- 고객 상세 정보
- 고객별 발행 이력

### 반드시 가능한 액션

- 새 고객 추가
- 고객 수정
- 팝빌 가입
- 팝빌 연결 초기화
- 팝빌 탈퇴
- 인증서 등록 URL 열기
- 인증서 상태 다시 확인
- 고객 삭제

### 상태/예외

- 고객 없음
- 검색 결과 없음
- 팝빌 미연결 고객 존재
- 인증서 미등록/만료 임박 고객 존재
- 발행 준비 완료 고객 존재
- 발행 준비 검토가 필요한 고객 존재

### 다른 기능과의 관계

- 발행 콘솔의 고객 문맥과 연결된다.
- 인증서 관리와 직접 연결된다.
- 초기 등록과 예외 메일 처리 결과가 이 화면의 데이터로 들어온다.

### 구조에 영향을 주는 사실

- 사용자는 여러 고객을 비교하면서 선택한다.
- 한 고객을 열어 세부 상태를 보고, 다시 다른 고객으로 이동하는 흐름이 잦다.
- 목록 맥락과 상세 맥락이 모두 중요하다.
- 생성/수정/위험 작업/이력 확인이 한 기능군 안에 같이 존재한다.

### 열어둘 쟁점

- 목록과 상세의 동시 노출 수준
- 신규 등록과 기존 고객 수정의 분리 수준
- 위험 작업 노출 방식

## 9. 인증서 관리

### 기능 목적

- 사용자가 로컬 공동인증서와 고객 연결 상태를 파악하고, 발급/갱신 준비 상태를 관리한다.

### 주요 사용자

- owner
- 작업공간 운영자

### 진입 맥락

- 도입 준비에서 인증서 단계로 진입한다.
- 운영 중에는 만료 임박, 미연결, 준비 필요 상태를 점검하려고 들어온다.

### 사용자가 완료하려는 일

- 현재 PC에서 읽힌 인증서를 확인한다.
- 인증서를 고객에 연결하거나 해제한다.
- 준비 필요/결제 가능 상태를 확인하고 후속 액션을 수행한다.

### 화면에 반드시 있어야 하는 정보

- 헬퍼 연결 상태
- helper 버전/업그레이드 필요 여부
- 로컬에서 읽힌 인증서 수
- 연결된 고객 수
- 미연결 인증서 수
- 준비 필요/결제 가능/만료 임박 상태
- 고객별 인증서 구성
- 미연결 인증서 후보 고객 정보

### 반드시 가능한 액션

- 헬퍼 상태 새로고침
- 로컬 인증서 다시 읽기
- 고객 인증서 연결
- 연결 해제
- 준비 실행
- 결제창 열기
- 고객/인증서 화면 간 이동

### 상태/예외

- helper 미설치
- helper 오프라인
- helper 업그레이드 필요
- 인증서 없음
- 미연결 인증서만 존재
- 고객 연결은 되었지만 준비 필요
- 결제 가능 상태 존재
- 만료 임박 인증서 존재

### 다른 기능과의 관계

- 고객 관리와 강하게 연결된다.
- 도입 준비의 초기 등록 후속 단계와 연결된다.
- 설정의 helper 상태와 연결된다.

### 구조에 영향을 주는 사실

- 이 화면은 `고객 상태 관점`과 `인증서 관점` 두 축을 동시에 가진다.
- 로컬 장치 상태와 서버에 저장된 연결 상태가 함께 보인다.
- 연결된 데이터와 미연결 데이터가 성격이 다르다.

### 열어둘 쟁점

- 고객 중심 관점과 인증서 중심 관점의 상대적 우선순위
- helper 상태와 인증서 실무 영역의 배치 비중
- 후속 액션 노출 수준

## 10. 설정 관리

### 기능 목적

- 작업공간 운영에 필요한 메일, 발행 기본값, helper, 계정/사용자 정보를 관리한다.

### 주요 사용자

- owner
- 일부 설정은 member가 볼 수 있지만, 사용자 관리 등은 owner 중심이다.

### 진입 맥락

- 도입 준비 중 설정 단계에서 들어온다.
- 운영 중 값 수정, 계정 관리, helper 점검 때문에 다시 들어온다.

### 사용자가 완료하려는 일

- 메일 연결을 확인한다.
- 발행 기본값을 입력한다.
- helper 상태를 확인한다.
- 본인 비밀번호나 작업공간 사용자를 관리한다.

### 화면에 반드시 있어야 하는 정보

- 현재 설정 섹션 목록
- 현재 섹션의 완료 여부
- 메일 연결 상태
- 발행 기본값 준비 상태
- helper 준비 상태
- 자동 저장 상태
- 사용자 목록과 권한 정보

### 반드시 가능한 액션

- 메일 연결 테스트
- 메일 관련 값 저장
- 발행 기본값 저장
- helper 상태 확인
- 사용자 추가
- 사용자 제거
- 사용자 임시 비밀번호 재설정
- 본인 비밀번호 변경

### 상태/예외

- 필수 설정 미입력
- 메일 테스트 진행 중/실패/성공
- 저장 중/저장 실패/저장 완료
- owner만 가능한 사용자 관리
- 저장된 비밀번호가 있지만 재표시되지 않는 상태

### 다른 기능과의 관계

- 도입 준비와 밀접하다.
- 인증서 관리와 helper 상태를 공유한다.
- owner 권한 흐름과 연결된다.

### 구조에 영향을 주는 사실

- 설정 안에서도 메일, 발행 기본값, helper, 계정 관리는 성격이 다르다.
- 일부 값은 자주 수정하지 않지만, 준비 여부는 자주 확인한다.
- 입력 폼과 상태 요약이 동시에 필요하다.
- 권한에 따라 가능한 액션이 갈린다.

### 열어둘 쟁점

- 설정 섹션의 상대적 독립성
- 상태 요약과 입력 폼의 배치 비중
- owner 전용 영역과 일반 사용자 영역의 구분 수준

## 11. 플랫폼 관리자 콘솔

### 기능 목적

- 플랫폼 관리자가 작업공간 개통, owner 계정 관리, 내부 작업, 포인트/자동화 상태를 운영한다.

### 주요 사용자

- platform admin

### 진입 맥락

- 일반 작업공간 운영이 아니라 서비스 전체 운영 목적으로 진입한다.

### 사용자가 완료하려는 일

- 새 작업공간을 만든다.
- 특정 작업공간 owner 비밀번호를 재설정한다.
- 작업공간 제한값을 조정한다.
- 내부 작업과 포인트/자동화 상태를 확인한다.

### 화면에 반드시 있어야 하는 정보

- 작업공간 목록
- 작업공간별 owner/상태/제한값/발행량
- owner 재설정 대상 정보
- 파트너 포인트 정보
- 내부 작업 실행 결과
- renewal agent heartbeat와 최근 진단 작업
- 운영 로그

### 반드시 가능한 액션

- 작업공간 생성
- managed customer limit 수정
- owner 임시 비밀번호 재설정
- internal jobs dispatch/run/maintenance 실행
- 파트너 충전 URL 열기
- agent 진단 액션 실행

### 상태/예외

- 플랫폼 정보 준비 안 됨
- 파트너 포인트 조회 실패
- 작업공간 없음
- 진단 작업 실패 존재
- 에이전트 오프라인

### 다른 기능과의 관계

- 일반 사용자 앱과 목적이 다르다.
- owner/작업공간 lifecycle을 통해 일반 앱과 연결된다.

### 구조에 영향을 주는 사실

- 사용자 앱의 일상 운영 정보와 플랫폼 운영 정보가 섞이면 목적이 흐려진다.
- 생성/수정 폼, 운영 상태 모니터링, 로그 조회가 한 화면 군에 같이 존재한다.
- 일반 사용자용 정보보다 예외와 운영 실패 신호가 중요하다.

### 열어둘 쟁점

- 생성 업무와 모니터링 업무의 분리 수준
- 사용자 앱과 관리자 앱의 시각적/구조적 거리감
- 로그/자동화/포인트 영역의 우선순위

## 기능 간 관계 요약

- `공개 로그인 -> 작업공간 초기 진입 -> 도입 준비/홈`
- `도입 준비 -> 설정 -> 초기 등록 -> 인증서 -> 첫 메일 동기화 -> 발행 확인`
- `홈 -> 발행 콘솔 / 고객 관리 / 도입 준비 / 설정`
- `고객 관리 <-> 인증서 관리`
- `발행 콘솔 <-> 고객 관리`
- `플랫폼 관리자`는 일반 사용자 앱과 목적이 다르며 owner/작업공간 관리 지점에서만 직접 맞닿는다.

## 와이어프레임 전 단계에서 특히 확인할 질문

- 어떤 기능은 `업무 실행`보다 `상태 판단`이 더 중요한가
- 어떤 기능은 `여러 항목 비교`가 핵심인가
- 어떤 기능은 `단계 진행`이 핵심인가
- 어떤 기능은 `예외 처리`가 정상 흐름만큼 중요한가
- 어떤 기능은 `사용자 앱`과 `운영자 앱`을 명확히 분리해서 봐야 하는가
- 어떤 기능은 `로컬 장치 상태`와 `서버 데이터 상태`를 같이 보여줘야 하는가
