# 초기 등록 기능 설계

## 목적

AUTO-TAX를 처음 도입하는 고객사는 이미 관리 중인 고객이 수십 곳에서 수백 곳까지 있을 수 있습니다.  
처음부터 고객을 하나씩 수기 등록하게 하면 도입 장벽이 너무 높아집니다.

이 문서는 `초기 등록` 기능의 MVP 범위를 정리합니다.

## 현재 제품 기준

- 고객 매칭은 `주소 기준`으로만 처리합니다.
- 고객 입력 화면에서는 `주소`를 하나만 받습니다.
- 저장 시 그 `주소`를 세금계산서 주소이자 메일 매칭 주소로 함께 사용합니다.
- `발전소명`은 고객 입력/매칭 기준에서 사용하지 않습니다.

즉 고객 등록에 필요한 핵심 필드는 아래 4개입니다.

- `대표자명`
- `사업자번호`
- `세금계산서 상호`
- `주소`

## 해결하려는 문제

### 1. 대량 고객 초기 등록

기존 고객이 100~300곳 이상인 경우, 현재 고객 등록 화면으로 하나씩 입력하는 방식은 비효율적입니다.

### 2. 엑셀 양식이 제각각

고객사마다 정리된 엑셀의 컬럼명과 순서가 다릅니다.  
고정 템플릿만 강제하면 실제 현장에서 쓰기 어렵습니다.

### 3. 이미 처리된 과거 메일 정리 필요

도입 시점 이전에 이미 발행이 끝난 월별 메일이 있을 수 있습니다.  
이 메일들을 현재 운영 건과 구분해서 정리할 수 있어야 합니다.

## 제품 방향

`초기 등록 센터`를 별도 탭으로 두고, 아래 기능을 순서대로 도입합니다.

1. `엑셀 컬럼 매핑 업로드`
2. `미등록 고객 빠른 등록`
3. `과거 메일 일괄 종료`

MVP는 1번부터 시작합니다.

## MVP: 엑셀 컬럼 매핑 업로드

### 지원 범위

- `xlsx`, `csv`
- 첫 시트만 지원
- 한 줄 = 한 고객 구조만 지원
- 필수 컬럼 4개만 매핑
  - `대표자명`
  - `사업자번호`
  - `세금계산서 상호`
  - `주소`

### 지원하지 않는 범위

- 병합셀 자동 해석
- 여러 시트 동시 처리
- 다중 헤더 구조 자동 인식
- 자유 형식 문서형 엑셀 완전 자동 해석

## 사용자 흐름

### 1. 파일 업로드

- 사용자가 `xlsx` 또는 `csv` 파일을 올립니다.
- 기본은 첫 시트를 읽습니다.

### 2. 헤더 행 선택

- `1행`, `2행`, `3행` 같은 헤더 후보를 보여줍니다.
- 사용자가 실제 헤더 행을 고릅니다.

### 3. 컬럼 매핑

- 파일의 컬럼명을 보여주고 아래 4개 필드에 연결합니다.
  - `대표자명`
  - `사업자번호`
  - `세금계산서 상호`
  - `주소`

### 4. 미리보기

- 상위 10행 정도를 보여줍니다.
- 원본값과 정리 결과를 같이 보여줍니다.
- 주소는 `신 도로명주소` 기준으로 정리된 값을 보여줍니다.

### 5. 검증

- 필수값 누락
- 사업자번호 숫자 정리
- 기존 고객과 사업자번호 중복
- 기존 고객과 주소 중복
- 주소 보정 실패

### 6. 가져오기

- 검증 통과 행만 저장합니다.
- 저장 기준은 아래와 같습니다.
  - `customerName = 대표자명`
  - `corpName = 세금계산서 상호`
  - `addr = 정리된 도로명주소`
  - `matchAddresses = [addr]`
  - `plantNames = []`

## 화면 구성

### 탭명

- `초기 등록`

### 섹션

1. `엑셀 업로드`
2. `컬럼 매핑`
3. `미리보기`
4. `검증 결과`
5. `가져오기 실행`

## 저장할 데이터

### 매핑 프로필

같은 고객사가 다음에도 비슷한 양식으로 업로드할 가능성이 크므로, 작업공간별 매핑을 저장합니다.

권장 테이블:

- `customer_import_profiles`
  - `id`
  - `organization_id`
  - `name`
  - `header_row_index`
  - `column_mapping_json`
  - `created_at`
  - `updated_at`

### 가져오기 작업 이력

MVP에서는 없어도 되지만, 운영 편의를 위해 이후 추가를 고려합니다.

- `customer_import_jobs`
- `customer_import_job_rows`

## API 설계

### 1. preview

`POST /api/customer-import/preview`

입력:

- 헤더 행 정보
- 컬럼 매핑 정보
- 원본 행 배열

출력:

- 정리된 행 데이터
- 행별 검증 결과
- 성공 가능 여부

### 2. commit

`POST /api/customer-import/commit`

입력:

- preview를 통과한 행 배열

처리:

- 기존 `saveCustomer` 재사용
- 주소는 `resolveRoadAddress` + `toRoadAddress` 기준으로 정리

출력:

- 성공 개수
- 실패 개수
- 실패 행과 사유

## 구현 방식

### 웹

- 브라우저에서 파일을 읽고 JSON으로 변환합니다.
- 서버에는 파일 자체가 아니라 파싱된 행 데이터와 매핑 정보만 보냅니다.

장점:

- 서버 업로드 처리 단순화
- `multer` 같은 추가 파일 업로드 계층이 필요 없음
- preview UI를 만들기 쉬움

### 서버

- 기존 고객 중복 검사
- 주소 정리
- 저장 전 검증
- 일괄 저장 처리

## 2차 기능

### 미등록 고객 빠른 등록

- 메일 파싱은 됐지만 고객 매칭이 안 된 건을 별도 큐에 표시
- 주소는 미리 채워둠
- 사용자는 `대표자명`, `사업자번호`, `세금계산서 상호`만 입력
- 저장 즉시 고객 등록 + 해당 메일 재매칭

### 과거 메일 일괄 종료

- 월별로 메일을 묶어서 표시
- 이미 발행 끝난 월은 `처리 완료`로 닫음
- 현재 초안 생성 대상에서 제외

## 구현 우선순위

1. `초기 등록` 탭 추가
2. `엑셀 업로드 + 컬럼 매핑`
3. `preview API`
4. `commit API`
5. `매핑 프로필 저장`
6. `미등록 고객 빠른 등록`
7. `과거 메일 일괄 종료`

## 결론

초기 등록 기능의 핵심은 `아무 엑셀을 완전 자동 해석`하는 것이 아니라,  
`필수 4개 필드만 사용자가 한 번 매핑하고 그 매핑을 저장해서 재사용`하는 데 있습니다.

이 방향이면 초기 도입 장벽을 크게 낮추면서도 구현 범위를 통제할 수 있습니다.
