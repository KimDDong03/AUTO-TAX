# AUTO-TAX Vercel + Supabase 전환 구조

## 1. 목적

이 문서는 AUTO-TAX를 앞으로 `웹 SaaS`로 운영할 때의 기본 인프라 방향을 정리하기 위한 문서이다.

이전 로컬 서버 + 로컬 파일 DB 중심 구조에서 벗어나, 아래 방향으로 전환한다.

- 프론트엔드 배포: Vercel
- 웹 API 배포: Vercel
- 데이터베이스: Supabase
- SaaS 운영 형태: 고객사별 데이터 분리 구조

## 2. 아키텍처 결정

AUTO-TAX의 목표 구조는 다음과 같다.

- 웹 화면은 Vercel에 배포한다.
- 고객사 데이터는 Supabase PostgreSQL에 저장한다.
- 로그인과 사용자 관리는 SaaS 기준으로 분리한다.
- 현재 로컬 파일 DB 기반 저장 방식은 제거한다.
- 현재 로컬 서버 내부의 상시 스케줄러 방식은 제거한다.

즉 앞으로는 `웹 SaaS + Supabase 기반 멀티테넌트 구조`를 기준으로 개발한다.

## 3. 왜 이 구조로 가는가

이 구조가 필요한 이유는 다음과 같다.

- 로컬 PC 설치형보다 SaaS가 고객사 확장에 유리하다.
- 태양광 회사 단위로 계정과 데이터를 분리해서 운영할 수 있다.
- Vercel은 웹 배포와 운영이 단순하다.
- Supabase는 PostgreSQL, 인증, 파일 저장 등 SaaS에 필요한 기본 요소를 함께 제공한다.
- 현재 단일 DB 구조는 단일 운영 기준이라 고객사가 여러 곳으로 늘어나면 관리가 어렵다.

## 4. 앞으로의 기본 역할 분리

### 4-1. Vercel

Vercel은 아래 역할을 맡는다.

- 웹 화면 제공
- API 요청 처리
- 로그인 이후 사용자 화면 제공
- 짧은 처리 시간의 서버 작업

현재 코드 기준 진입점은 아래와 같다.

- Vercel Express 함수 진입점: `src/server.ts`
- 정적 웹 산출물: `public`
- 배포 설정: `vercel.json`

### 4-2. Supabase

Supabase는 아래 역할을 맡는다.

- 고객사 데이터 저장
- 사용자 계정 및 권한 관리
- 발행 이력 및 로그 저장
- 고객별 설정 저장
- 파일 저장이 필요할 경우 첨부 자료 또는 산출물 저장

### 4-3. 별도 처리 대상

아래 작업은 Vercel에 그대로 두면 안 된다.

- 계속 돌아가야 하는 메일 폴링
- 장시간 실행되는 일괄 작업
- 공동인증서 갱신처럼 특정 PC 환경이 필요한 작업

이 작업들은 별도 작업 구조로 빼야 한다.

## 5. Vercel에서 바로 안 맞는 현재 구조

이전 코드에서 Vercel에 바로 맞지 않았던 부분은 다음과 같다.

### 5-1. 로컬 파일 DB 사용

이전에는 로컬 파일 DB를 직접 읽고 썼다.

Vercel에서는 이 구조를 운영 기준으로 사용할 수 없다.

앞으로는 Supabase PostgreSQL 테이블 구조로 바꿔야 한다.

### 5-2. 서버 안의 상시 스케줄러

이전에는 서버 안에서 `setInterval` 방식으로 메일 동기화와 자동 발행을 처리했다.

Vercel은 이런 식의 상시 실행 서버 구조와 맞지 않는다.

즉 현재 로컬 실행에서는 돌아가는 작업도, Vercel 배포만으로는 같은 방식으로 계속 돌지 않는다.

앞으로는 다음 중 하나로 바꿔야 한다.

- Vercel Cron으로 정해진 시간마다 작업 호출
- Supabase 쪽 작업 큐 기반 처리
- 별도 워커 서버에서 백그라운드 작업 처리

### 5-3. 로컬 파일 전제

이전에는 아래처럼 로컬 파일 기준 요소가 존재했다.

- 로컬 DB 파일
- DB 백업 파일
- 로컬 임시 파일

SaaS 구조에서는 이 전제를 줄이거나 없애야 한다.

### 5-4. 단일 운영자 기준 설정 구조

이전 설정은 앱 전체 공통값에 가까웠다.

SaaS 구조에서는 아래처럼 바뀌어야 한다.

- 고객사별 설정
- 고객사별 사용자
- 고객사별 메일 연동 정보
- 고객사별 팝빌 정보
- 고객사별 운영 정책

## 6. 목표 데이터 구조

Supabase 기준으로는 대략 아래 구조가 필요하다.

### 6-1. 고객사

AUTO-TAX를 구독하는 태양광 회사

예상 항목:

- 고객사명
- 사업자번호
- 상태
- 요금제
- 생성일

### 6-2. 고객사 사용자

고객사 내부 사용자 계정

예상 항목:

- 고객사 ID
- 사용자 ID
- 이름
- 이메일
- 권한

### 6-3. 관리 고객

고객사가 실제로 관리하는 발행 대상 고객

예상 항목:

- 고객사 ID
- 사업자 정보
- 발전소명
- 발행 설정
- 팝빌 상태
- 인증서 상태

### 6-4. 메일 수집 이력

메일 원문, 처리 상태, 연결 결과를 저장

### 6-5. 발행 초안 / 발행 이력

초안 생성, 검수, 발행 완료, 실패 이력을 저장

### 6-6. 인증서 상태

고객별 공동인증서 상태, 만료일, 갱신 필요 여부를 저장

### 6-7. 운영 로그

고객사 기준 작업 로그, 오류 로그, 감사 로그를 저장

## 7. 권장 서비스 구조

현실적인 구조는 아래와 같다.

### 7-1. 웹 앱

- Vercel에 배포
- 고객사 로그인
- 관리 고객 관리
- 발행 관리
- 인증서 상태 확인

### 7-2. API

- Vercel 서버 함수 또는 API 라우트
- 고객사별 데이터 조회/수정
- 초안 생성 요청
- 발행 요청
- 관리 화면용 데이터 제공

### 7-3. 데이터베이스

- Supabase PostgreSQL 사용
- 고객사 단위 데이터 분리
- 권한 정책 적용

### 7-4. 백그라운드 작업

아래 작업은 별도로 운영하는 것이 안전하다.

- 메일 동기화 작업
- 자동 발행 작업
- 인증서 만료 점검
- 알림 발송 배치

## 8. 공동인증서 갱신 기능의 위치

공동인증서 갱신은 일반 웹 배포만으로 끝나는 기능이 아니다.

그 이유는 다음과 같다.

- 공동인증서 갱신은 특정 PC 환경 영향을 받는다.
- 보안 프로그램이 필요한 경우가 있다.
- 브라우저만으로 처리하기 어려운 흐름이 있다.

따라서 이 기능은 여전히 별도 구조로 보는 것이 맞다.

권장 방향:

- 웹 SaaS에서 상태와 요청을 관리
- 실제 갱신 보조는 별도 로컬 에이전트 또는 전용 PC에서 처리

즉 `웹 SaaS + 로컬 갱신 보조 구조`로 가는 것이 현실적이다.

## 9. 바로 바꿔야 하는 코드 방향

앞으로 코드 전환은 아래 순서가 적절하다.

1. 기존 `Store` 구조 제거 준비
2. Supabase용 데이터 모델 설계
3. 고객사 / 관리 고객 / 사용자 권한 구조 분리
4. 현재 Express API를 Vercel 배포 가능한 API 구조로 재구성
5. 서버 내부 스케줄러 제거
6. 메일 동기화와 자동 발행을 작업 큐 또는 크론 구조로 분리
7. 인증서 갱신 보조 기능은 별도 워커 구조로 유지

## 10. 제품 용어 통일

이 구조로 가면 아래 용어를 기준으로 문서를 맞춰야 한다.

- `고객사`: AUTO-TAX를 구독하는 태양광 회사
- `사용자`: 고객사 내부 운영자
- `관리 고객`: 고객사가 관리하는 실제 발행 대상 고객
- `작업공간`: 고객사별 독립 데이터 공간
- `발행 건`: 관리 고객 기준 세금계산서 발행 단위

## 11. 결론

Vercel + Supabase 방향은 AUTO-TAX를 SaaS로 키우는 데 맞는 선택이다.

다만 중요한 점은 `웹으로 배포만 바꾸는 것`이 아니라, 현재 로컬 설치형 구조를 `멀티테넌트 SaaS 구조`로 다시 설계해야 한다는 점이다.

즉 앞으로의 핵심 전환 포인트는 다음 3가지다.

- 로컬 파일 DB 제거
- 상시 스케줄러 제거
- 고객사별 데이터/권한 구조 도입

이 원칙을 기준으로 이후 개발 방향을 잡는다.
