# AUTO-TAX 출시 전 체크리스트

이 문서는 `실제 고객사에 도입하고 유료 운영을 시작하기 전에` 반드시 확인해야 할 항목을 빠르게 점검하기 위한 문서다.

기준:

- 목표는 `기능 추가`가 아니라 `실전 운영 가능 상태 확인`이다.
- 최소 1개 실제 고객사 또는 실제와 거의 같은 테스트 고객사 기준으로 끝까지 리허설한다.
- 체크는 `운영 환경` 기준으로 한다. 로컬 확인만으로 통과 처리하지 않는다.

## 2. 첫 고객 리허설

첫 고객사 기준으로 아래 흐름을 끝까지 확인한다.

1. 고객사 작업공간 개통
2. owner 로그인 전달
3. 메일 계정 입력
4. 고객 1명 이상 등록
5. 발전소명 등록
6. 팝빌 가입
7. 인증서 등록 상태 확인
8. 테스트 메일 1건 동기화
9. 초안 1건 생성 확인
10. 수동 발행 1건 성공 확인
11. 실패 케이스 1건 의도적으로 만들어 로그/알림 확인
12. 자동 발행 고객 1곳으로 월 자동 처리 점검

리허설 통과 기준:

- 메일 수집부터 발행까지 사람 손으로 우회하지 않고 시스템 흐름 안에서 끝난다.
- 운영자가 실패 원인을 로그와 알림에서 바로 파악할 수 있다.
- 고객사가 봐야 하는 화면과 운영자가 봐야 하는 화면이 명확히 구분된다.

## 3. 과금/영업 운영 확정

실제 유료 운영 전 아래 기준을 문서로 고정한다.

- [ ] 오픈베타 요금 확정
- [ ] 정식 요금 확정
- [ ] `관리 고객 수` 계산 기준 확정
- [ ] 언제 기준으로 고객 수를 집계하는지 확정
- [ ] 초과 과금 반영 시점 확정
- [ ] 오픈베타 -> 정식 전환 날짜 확정
- [ ] 문의 접수 후 응답 방식 확정
- [ ] 작업공간 개통 절차 확정

권장 고정 항목:

- 관리 고객 수는 `해당 월 활성 고객 기준`으로 센다.
- 초과 과금은 `다음 결제 주기`에 반영한다.
- 예외 처리나 별도 운영 지원은 기본 구독과 분리한다.

## 9. 아직 출시 전 막히면 안 되는 항목

아래 중 하나라도 불확실하면 유료 운영 시작을 미루는 것이 맞다.

- 실제 고객사 기준 메일 -> 매칭 -> 초안 -> 발행 흐름 미검증
- 자동 배치가 운영 환경에서 실제로 안 돌아감
- 실패 알림이 운영자에게 안 감
- owner 개통/로그인 흐름이 불안정함
- 과금 기준이 아직 말로만 있고 문서로 고정되지 않음

## 10. 최종 판단 기준

`출시 가능` 기준:

- 운영 환경에서 핵심 흐름이 재현된다.
- 실제 고객사 1곳 기준 리허설이 끝났다.
- 장애 시 누가 무엇을 확인할지 정해져 있다.
- 가격과 개통 절차를 바로 설명할 수 있다.

`출시 보류` 기준:

- 핵심 흐름을 아직 로컬에서만 확인했다.
- 실제 실패 케이스 대응을 아직 안 해봤다.
- 첫 고객사 온보딩 절차가 아직 손에 익지 않았다.
