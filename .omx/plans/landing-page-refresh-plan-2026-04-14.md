# AUTO-TAX 랜딩 페이지 개편 계획

작성일: 2026-04-14

## Requirements Summary

- 현재 공개 랜딩은 `web/src/App.tsx:5975-6215`에 인라인으로 들어 있으며, 실질적으로 **히어로 + 로그인/문의 + 가격**만 노출된다.
- 공개 섹션 anchor도 `landing-top`, `landing-login-card`, `landing-pricing` 3개뿐이라(`web/src/App.tsx:6002`, `6032`, `6133`) 정보 밀도가 낮고 완성도가 떨어져 보이기 쉽다.
- 디자인 기준상 AUTO-TAX는 화려한 SaaS 랜딩보다 **운영 도구처럼 차분하고 신뢰감 있게** 보여야 하며(`DESIGN.md:3`), **마케팅 랜딩 느낌 / 카드 중첩 / 과한 장식**은 피해야 한다(`DESIGN.md:11`, `15`, `177`, `182`).
- 현재 랜딩용 스타일에는 `landing-fit-section`, `landing-faq-grid` 등 확장 가능한 섹션 토대가 이미 있으나(`web/src/styles.css:848`, `886`), 실제 마크업은 없다.

## Goal

“싸구려 SaaS 랜딩”이 아니라 **실무형 운영 도구 소개 페이지**처럼 보이도록 구조와 카피를 재정리한다.

## Acceptance Criteria

1. 첫 화면만 보고도 **누구를 위한 제품인지 / 무엇을 자동화하는지 / 어떤 방식으로 운영되는지** 이해된다.
2. 로그인 영역이 메인 메시지를 방해하지 않고, **도입 문의**와 **로그인**의 목적이 분리되어 읽힌다.
3. 가격 이전에 **업무 흐름 / 적합 고객 / 신뢰 근거**가 제시된다.
4. 전체 공개 랜딩이 최소 **4개 정보 블록**(히어로, 운영 흐름/가치, 가격, FAQ 또는 적합 대상)으로 구성된다.
5. 시각 톤이 `DESIGN.md`의 운영 도구 원칙과 어긋나지 않는다.

## Recommended Direction

### 방향
**“영업용 랜딩”보다 “운영 전환 안내 페이지”**로 재구성한다.

### 핵심 톤
- 감성/과장 대신: 정확함, 반복 업무 절감, 검수 중심 운영
- 추상 카피 대신: 한전 메일 확인, 초안 생성, 검수 후 발행, 고객 수 기준 과금
- 화려한 장식 대신: 상태/흐름/근거 중심 블록

## Main Problems in Current Page

1. **정보 구조가 너무 얕다**
   - 공개 섹션이 사실상 2개라 페이지가 “덜 만든 느낌”을 준다. (`web/src/App.tsx:6002-6215`)
2. **로그인 카드가 너무 빨리 튀어나온다**
   - 첫 화면 우측에서 로그인/문의가 메인 메시지와 경쟁한다. (`web/src/App.tsx:6031-6130`)
3. **가격이 너무 앞에 나온다**
   - 왜 필요한지 설득하기 전에 가격부터 보이므로 값어치 인지가 약하다. (`web/src/App.tsx:6133-6215`)
4. **제품의 운영 방식이 충분히 시각화되지 않는다**
   - `LANDING_HERO_POINTS` 3개만으로는 실제 워크플로를 전달하기 부족하다. (`web/src/App.tsx:586-600`)
5. **구조 대비 스타일 자산 활용이 덜 됐다**
   - CSS에는 fit / faq / step 계열 토대가 있으나 실제 섹션이 비어 있다. (`web/src/styles.css:495-500`, `848-886`)

## Implementation Steps

### 1. 공개 랜딩의 메시지 구조를 먼저 다시 짠다
- 대상 파일: `web/src/App.tsx`
- 작업:
  - 랜딩을 최소 4개 블록으로 재구성한다.
  - 추천 순서:
    1. Hero
    2. 업무 흐름 / 핵심 가치
    3. 적합한 고객 / 운영 방식
    4. 가격
    5. FAQ + 문의 CTA
- 메모:
  - 가격은 유지하되, 설득 이후에 나오게 뒤로 보낸다.

### 2. 히어로를 “한 줄 가치 + 운영 근거” 구조로 바꾼다
- 대상 파일: `web/src/App.tsx:6002-6030`, `web/src/styles.css:419-566`
- 작업:
  - 현재 `h1`/설명/버튼/3개 proof 카드 구조는 유지하되, 메시지를 더 선명하게 정리한다.
  - 추천 카피 구조:
    - 상단 badge: `태양광 전자세금계산서 운영 도구`
    - 제목: `한전 메일 확인부터 세금계산서 초안·검수·발행까지 한 흐름으로`
    - 보조문: `반복 확인 작업은 줄이고, 담당자는 검수와 예외 처리에 집중합니다.`
  - proof 카드는 “기능 소개”보다 “운영 결과” 관점으로 바꾼다.
    - 예: `메일 확인 자동화`, `초안 준비 단축`, `검수 후 발행 유지`

### 3. 로그인/문의 영역을 “보조 패널”로 낮춘다
- 대상 파일: `web/src/App.tsx:6031-6130`
- 작업:
  - 우측 카드의 1순위 목적을 `도입 문의`가 아니라 `기존 고객 로그인`으로 분리해 명확히 한다.
  - 문의 폼은 기본 접힘 상태 유지하되, 클릭 시 펼쳐지는 현재 패턴은 유지 가능.
  - 다만 카드 헤드라인을 `도입 문의와 로그인`에서 아래처럼 분리한다:
    - 제목: `기존 고객 로그인`
    - 보조문: `도입 상담이 필요하면 아래 문의를 남겨주세요.`
  - 문의 CTA는 hero 하단과 FAQ 하단에도 한 번 더 배치한다.

### 4. “어떻게 동작하는지” 섹션을 추가한다
- 대상 파일: `web/src/App.tsx`, 기존 스타일 후보 `web/src/styles.css:495-500`, `625-641`
- 작업:
  - 새 섹션 예시: `업무 흐름`
  - 3~4단계 step 카드로 구성:
    1. 한전 메일 수집
    2. 발행 대상 정리
    3. 세금계산서 초안 생성
    4. 검수 후 발행
  - 목적:
    - 제품 소개를 추상 설명이 아니라 실제 운영 흐름으로 보여준다.

### 5. “이런 회사에 맞습니다” 섹션을 추가한다
- 대상 파일: `web/src/App.tsx`, 기존 스타일 후보 `web/src/styles.css:848-884`
- 작업:
  - `landing-fit-section` / `landing-fit-card`를 실제 마크업에 연결한다.
  - 예시 항목:
    - 관리 고객사가 일정 수 이상 쌓여 있는 태양광 사업자
    - 매달 메일 확인과 발행 준비가 반복되는 팀
    - 완전 자동보다 `검수 후 발행` 통제를 원하는 운영팀
  - 이 블록이 있으면 “누구용인지”가 명확해진다.

### 6. 가격 섹션은 유지하되, 더 “운영형”으로 보정한다
- 대상 파일: `web/src/App.tsx:6133-6215`, `web/src/styles.css:643-846`
- 작업:
  - 현재 계산기는 장점이 있으므로 유지한다.
  - 대신 가격 설명 상단에 짧은 운영 기준 문구를 추가한다.
    - 예: `관리 고객 수를 기준으로 비용을 예측할 수 있도록 단순한 월 구독 구조를 사용합니다.`
  - 가격 카드 badge 명칭은 더 덜 마케팅스럽게 다듬는다.
    - `OPEN BETA`, `STANDARD`보다 한국어 운영형 명칭 검토

### 7. FAQ 또는 신뢰 근거 블록으로 하단 마무리를 만든다
- 대상 파일: `web/src/App.tsx`, 기존 스타일 후보 `web/src/styles.css:886-...`
- 작업:
  - FAQ 3~4개만 둔다.
  - 추천 질문:
    - 어떤 회사가 쓰는 도구인가요?
    - 자동 발행만 가능한가요?
    - 고객 수가 늘면 요금은 어떻게 계산되나요?
    - 도입 전에 상담할 수 있나요?
  - 마지막 CTA는 `도입 문의 남기기` 하나로 정리한다.

### 8. 랜딩 UI를 App.tsx에서 분리한다
- 대상 파일: `web/src/App.tsx`
- 신규 후보:
  - `web/src/features/public/PublicLanding.tsx`
  - 필요 시 `web/src/features/public/public-content.ts`
- 작업:
  - 공개 랜딩 블록이 커질수록 `App.tsx` 인라인 유지가 더 빨리 “짜친다”.
  - 섹션 데이터, 카피, CTA를 분리해 이후 수정 비용을 낮춘다.

## Suggested Copy Principle

- 짧고 실무형으로 쓴다.
- “혁신”, “스마트”, “압도적”, “간편하게” 같은 추상 표현은 피한다.
- 기능명보다 운영 결과를 앞에 둔다.

## Priority Order

1. 히어로 카피/구조 재작성
2. 로그인 카드 톤 다운
3. 업무 흐름 섹션 추가
4. 적합 대상 섹션 추가
5. 가격 섹션 문구 보정
6. FAQ/하단 CTA 추가
7. 컴포넌트 분리

## Risks and Mitigations

- **위험:** 너무 랜딩답게 꾸미면 제품 디자인 원칙과 충돌
  - **대응:** `DESIGN.md` 기준대로 상태/흐름/운영 언어 중심 유지
- **위험:** 로그인 사용자의 즉시 진입성이 떨어질 수 있음
  - **대응:** 상단 `로그인` 버튼은 유지하고, 우측 로그인 카드도 남기되 위계를 낮춘다
- **위험:** 섹션을 늘리며 App.tsx가 더 비대해질 수 있음
  - **대응:** 공개 랜딩은 별도 feature로 분리

## Verification Steps

2. 모바일에서 hero → 문의/로그인 → 흐름 → 가격 순서가 자연스럽게 내려가는지 확인
3. 기존 로그인 동선이 그대로 동작하는지 확인
4. 가격 계산기 입력/플랜 전환이 유지되는지 확인
5. 카피가 `DESIGN.md`의 운영 도구 톤과 충돌하지 않는지 점검
